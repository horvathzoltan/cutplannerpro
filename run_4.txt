**Rövid, lényegi válasz:**
A SeriesMatrixView jelenlegi implementációja *minden frissítéskor teljes mátrix‑újraépítést végez*, ami **O(N×M)** komplexitású (N = BOM anyagok száma, M = tételszámok száma). Ez a leglassabb lehetséges megoldás.
A gyorsítás kulcsa: **incrementális frissítés**, **cache‑elt BOM**, **cache‑elt filledCells**, **új cellák csak akkor jönnek létre, ha szükséges**, és **nem rebuildeljük a teljes táblát minden alkalommal**.

Az alábbi terv **pontokba szedve**, auditálható módon mutatja meg, hogyan lehet a mátrixot 10–30× gyorsabbá tenni úgy, hogy **QTableWidget marad**, tehát nincs model/view, nincs SQL‑veszély, nincs pointer‑nullás adatvesztés.

---

# ⭐ 1. A jelenlegi lassúság okai (diagnózis)
A SeriesMatrixView minden frissítéskor:

- újraolvassa a registryt
- újraépíti a BOM‑ot
- újraépíti a filledCells‑et
- újraépíti az összes sort
- újraépíti az összes oszlopot
- újraépíti az összes cellát
- minden cellába új QTableWidgetItem kerül
- minden cella új háttérszínt kap
- minden cella új szimbólumot kap
- minden cella új UserRole adatot kap

Ez **több ezer objektum** újraallokálása minden egyes frissítéskor → lassú.

---

# ⭐ 2. A gyorsítás fő iránya: incrementális frissítés
A mátrixot **nem kell teljesen újraépíteni** minden request vagy anyag hozzáadásakor.

### A helyes működés:
- **Új request → új oszlop**
- **Új anyag → egy cella frissítése**
- **Request módosítás → csak az adott oszlop cellái frissülnek**
- **BOM változás → csak a sorok frissülnek**

Ez a mátrixot **nagyságrendekkel gyorsabbá** teszi.

---

# ⭐ 3. BOM cache (óriási gyorsulás)
Most minden rebuild:

```cpp
generateBomMaterials(req)
```

Ez BOM‑számítás:

- MaterialRegistry bejárása
- BOM családok bejárása
- prefixek bejárása
- színszűrés
- stabil sorrend generálása

Ez brutális.

### Megoldás:
```cpp
QHash<QPair<typeId, subtypeId>, QVector<QUuid>> _bomCache;
```

Használat:

```cpp
auto key = qMakePair(req.productTypeId, req.productSubtypeId);
if (_bomCache.contains(key))
    return _bomCache[key];

auto bom = generateBomMaterials(req);
_bomCache[key] = bom;
return bom;
```

### BOM cache csak akkor frissül:
- új request
- request módosítás
- request törlés

Ez **5–10× gyorsulás**.

---

# ⭐ 4. filledCells cache
Most minden rebuild:

```cpp
for (const auto& r : all)
    _active.filledCells.insert({ r.externalReference, r.materialId });
```

Ez minden cellát újra generál → lassú.

### Megoldás:

```cpp
QSet<QPair<QString, QUuid>> _filledCache;
```

Frissítés:

- új request → új pár
- request módosítás → régi pár törlése, új pár hozzáadása
- request törlés → pár törlése

Ez **nagyon sok időt spórol**.

---

# ⭐ 5. Sorok és oszlopok csak akkor frissüljenek, ha tényleg változnak

### Oszlopok:
- csak akkor nő a tételszám lista, ha új request jön
- tehát csak akkor kell `insertColumn()`

### Sorok:
- BOM csak akkor változik, ha új productType/subtype jön
- tehát csak akkor kell `insertRow()`

Ez **2× gyorsulás**.

---

# ⭐ 6. Cellák csak akkor jönnek létre, ha szükséges
Most minden cella új QTableWidgetItem:

```cpp
auto* item = new QTableWidgetItem(symbol);
```

Ez lassú.

### Megoldás:

```cpp
auto* item = _table->item(row, col);
if (!item) {
    item = new QTableWidgetItem();
    _table->setItem(row, col, item);
}
item->setText(symbol);
item->setBackground(bg);
item->setData(Qt::UserRole, ...);
```

Ez **nagyon gyors**.

---

# ⭐ 7. Incrementális frissítés algoritmusa (konkrét terv)

## Új request hozzáadása
1. `_full.append(newRequest)`
2. új oszlop index = `_table->columnCount()`
3. `_table->insertColumn(newCol)`
4. `_table->setHorizontalHeaderItem(newCol, new QTableWidgetItem(ref))`
5. minden BOM sorra:
   - `updateCell(row, newCol)`

## Request módosítása
1. BOM cache frissítése
2. filledCells cache frissítése
3. az adott oszlop celláinak frissítése:
   - `updateCell(row, col)`

## Új anyag hozzáadása
1. filledCells cache frissítése
2. csak az adott cella frissítése:
   - `updateCell(rowIndexOfMaterial, colIndexOfReference)`

## BOM változás
1. új sorok hozzáadása
2. minden oszlopra:
   - `updateCell(newRow, col)`

---

# ⭐ 8. A végső implementációs / patching terv

### 1) BOM cache bevezetése
- `_bomCache` tagváltozó
- `generateBomMaterials()` módosítása cache‑re

### 2) filledCells cache bevezetése
- `_filledCache` tagváltozó
- frissítés minden add/update/delete műveletnél

### 3) `rebuildMatrix()` szétbontása
- `buildHeadersIncremental()`
- `buildRowsIncremental()`
- `updateCell(row, col)`

### 4) `refreshAfterAdd()` módosítása
- csak új oszlopot adjon hozzá
- ne rebuildeljen teljes mátrixot

### 5) `setActiveReference()` módosítása
- csak kijelölés → ne rebuildeljen

### 6) `onSeriesContextChanged()` módosítása
- csak kijelölés → ne rebuildeljen

### 7) `updateMatrix()` teljes refaktor
- csak akkor rebuildeljen, ha BOM vagy tételszám lista változott

### 8) Teljes mátrix rebuild csak akkor:
- ha BOM változik
- ha tételszám lista változik
- ha user explicit újratöltést kér

---
